1. System Architecture

The system is built on a Selective-Repeat ARQ (Automatic Repeat reQuest) architecture over UDP. Given the strict bandwidth cap (≤ 2.0x), continuous Forward Error Correction (FEC) via packet duplication was mathematically unviable (resulting in >2.05x overhead). Therefore, the system utilizes a feedback loop where the receiver explicitly requests retransmissions (NACKs) only for dropped packets.

2. Design Choices & System Level Implementation

  • Non-Blocking I/O: Both the sender and receiver utilize poll() for multiplexed, non-blocking I/O. The sender listens concurrently to the harness (port 47010) and the feedback lane (port 47004) to ensure NACKs are processed instantly without halting the main media stream.
  • Zero-Buffer Playout: Exploiting the harness player's tolerance for out-of-order packets (provided they arrive before the deadline), the receiver eliminates the standard jitter buffer. Incoming packets are immediately forwarded to port 47020, radically minimizing base playout delay.
  • Aggressive NACK Strategy: The receiver tracks a trailing window of sequence numbers. A high-frequency polling loop checks for missing sequence numbers and fires lightweight (4-byte) NACKs to the sender if a packet is missing and its theoretical deadline has not yet passed.
  • Bandwidth Optimization: By transmitting only 164-byte original packets and 4-byte NACKs, the system easily maintains an overhead of ~1.1x, leaving ample room to aggressively spam NACKs in high-jitter environments without risking disqualification.
